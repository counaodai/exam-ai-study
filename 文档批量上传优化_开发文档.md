# 文档批量上传优化 — 开发文档

> **版本**: v1.0  
> **日期**: 2026-05-28  
> **对应需求**: 优化需求文档 → 三、需求二：文档批量上传优化

---

## 一、问题分析

### 1.1 原有实现的问题

| 问题 | 根因 |
|------|------|
| 仅支持单文件逐一上传 | `el-upload` 组件未启用 `multiple` 属性 |
| 无上传队列管理 | 每次上传独立处理，无统一状态跟踪 |
| 无并发控制 | 多文件同时上传会导致服务器压力过大 |
| 无文件夹上传支持 | 缺少 `webkitdirectory` 属性和文件夹遍历逻辑 |
| 无批量错误处理 | 单个文件失败不影响其他文件，但也无法重试 |

### 1.2 原有数据流

```
用户选择单个文件 → handleUpload() → documentStore.uploadDocument() → API POST → 等待完成 → 显示成功
```

---

## 二、技术方案

### 2.1 整体架构

采用**前端队列 + 并发控制**实现批量上传：

```
用户选择多个文件/文件夹 → addToQueue() → 队列管理 → 最多3个并发上传 → 实时进度跟踪 → 完成/失败处理
```

### 2.2 核心设计

| 设计点 | 实现方式 |
|--------|----------|
| 多文件选择 | `el-upload` 组件的 `multiple` 属性 |
| 文件夹上传 | 动态创建 `<input webkitdirectory>` 元素 |
| 拖拽上传 | `el-upload` 组件的 `drag` 属性 |
| 并发控制 | 最多 3 个同时上传（`MAX_CONCURRENT = 3`） |
| 队列管理 | Pinia store 中的 `uploadQueue` 响应式数组 |
| 进度跟踪 | Axios 的 `onUploadProgress` 回调 |
| 错误重试 | 最多 2 次自动重试（`MAX_RETRY = 2`） |

### 2.3 为什么使用前端队列而非后端批量接口

- 后端现有单文件上传接口无需改动，前端循环调用即可
- 每个文件独立处理，失败不影响其他文件
- 可以实时跟踪每个文件的上传进度
- 并发控制在前端实现更灵活

### 3.1 后端接口保持不变

原有 `POST /api/documents/upload` 接口无需改动，前端循环调用即可。

### 3.2 前端 API 变更

**`api/document.ts`** — `uploadDocument` 函数新增进度回调参数：

```typescript
export interface UploadProgressCallback {
  onProgress?: (percent: number) => void
}

export function uploadDocument(
  file: File,
  module?: string,
  callbacks?: UploadProgressCallback,
) {
  // ... 使用 Axios 的 onUploadProgress 回调
}
```

超时时间从 120 秒增加到 300 秒，以支持大文件上传。

## 四、核心实现逻辑

### 4.1 类型定义

```typescript
export type UploadQueueStatus = 'waiting' | 'uploading' | 'completed' | 'failed' | 'cancelled'

export interface UploadQueueItem {
  id: string
  file: File
  filename: string
  file_size: number
  status: UploadQueueStatus
  progress: number
  error: string | null
  retryCount: number
}
```

### 4.2 队列管理 Store

**文件: `frontend/src/stores/document.ts`**

核心状态：

```typescript
const uploadQueue = ref<UploadQueueItem[]>([])
const MAX_CONCURRENT = 3
const MAX_RETRY = 2
```

核心方法：

| 方法 | 功能 |
|------|------|
| `addToQueue(files)` | 将文件数组添加到队列，触发队列处理 |
| `processQueue()` | 检查并发限制，启动待上传文件 |
| `uploadSingleItem(item)` | 上传单个文件，跟踪进度，处理成功/失败 |
| `cancelQueueItem(id)` | 取消单个队列项 |
| `retryQueueItem(id)` | 重试失败的队列项 |
| `cancelAllQueue()` | 取消所有等待中和上传中的队列项 |
| `clearFinishedQueue()` | 清除已完成的队列项 |

队列处理流程：

```
addToQueue(files)
├── 创建 UploadQueueItem 数组
├── 添加到 uploadQueue
└── 调用 processQueue()

processQueue()
├── 检查 uploading 状态
├── 计算可用槽位 (MAX_CONCURRENT - 当前上传数)
├── 选择等待中的文件
└── 并行调用 uploadSingleItem()

uploadSingleItem(item)
├── 设置状态为 uploading
├── 调用 API 上传（带进度回调）
├── 成功：更新状态为 completed，添加到文档列表
└── 失败：重试或标记为 failed
```

### 4.4 文件夹上传实现

```typescript
function triggerFolderUpload() {
  const input = document.createElement('input')
  input.type = 'file'
  input.setAttribute('webkitdirectory', '')
  input.setAttribute('directory', '')
  input.multiple = true
  input.accept = ALLOWED_EXTENSIONS.join(',')
  input.style.display = 'none'
  input.addEventListener('change', handleFolderSelected)
  document.body.appendChild(input)
  input.click()
}

function handleFolderSelected(event: Event) {
  const input = event.target as HTMLInputElement
  const files: File[] = []
  for (let i = 0; i < input.files.length; i++) {
    const file = input.files[i]
    if (isAllowedFile(file.name)) {
      files.push(file)
    }
  }
  documentStore.addToQueue(files)
}
```

关键点：
- 使用 `webkitdirectory` 属性支持文件夹选择
- 自动过滤不支持的文件格式
- 显示过滤结果提示

## 六、异常处理机制

| 场景 | 处理方式 |
|------|----------|
| 单个文件上传失败 | 自动重试最多 2 次，重试失败后标记为失败 |
| 网络中断 | Axios 请求超时（300 秒），捕获异常并标记失败 |
| 用户取消上传 | 设置队列项状态为 `cancelled`，跳过后续处理 |
| 文件格式不支持 | 文件夹上传时自动过滤，显示跳过数量提示 |
| 所有文件失败 | 队列显示失败状态，用户可逐个重试 |
