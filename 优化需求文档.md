## 二、需求一：AI 问答流式输出优化

### 2.1 问题详细描述

**当前行为**：

```
用户发送问题 → 等待 AI 完整生成回答（可能需要 10-30 秒） → 一次性展示全部内容
```

**问题影响分析**：

| 影响维度  | 具体表现                           |
| ----- | ------------------------------ |
| 用户体验  | 等待期间无任何反馈，用户无法判断系统是否在正常工作      |
| 交互流畅性 | 长回答（如解题步骤、知识点解析）等待时间过长，阅读节奏被打断 |
| 心理感受  | "卡顿感"明显，用户可能误以为系统故障而重复提交       |

### 2.2 核心需求描述

将 AI 问答的回答输出模式从「批量输出」改为「流式输出（Streaming）」。

**目标行为**：

```
用户发送问题 → AI 开始生成 → 逐字/逐词实时展示在界面上 → 生成完毕
```

**技术实现要点**：

| 层级    | 实现要求                                                                                      |
| ----- | ----------------------------------------------------------------------------------------- |
| 后端    | 使用 SSE（Server-Sent Events）实现流式响应，通过 FastAPI 的 `StreamingResponse` 或 `EventSourceResponse` |
| 前端    | 使用 EventSource 或 Fetch API 的 ReadableStream 接收流式数据                                        |
| UI 展示 | 文字逐字拼接到现有内容后方，保持 Markdown 格式正确渲染                                                          |
| 完成判断  | 需有明确的流结束信号，触发「发送」按钮恢复可用状态                                                                 |

### 2.3 验收标准

| 编号   | 验收项           | 通过条件                                  |
| ---- | ------------- | ------------------------------------- |
| AC-1 | 流式输出基础能力      | AI 回答以逐字拼接形式实时展示，而非等待完成后一次性显示         |
| AC-2 | Markdown 实时渲染 | 流式输出过程中，Markdown 格式（标题、加粗、列表等）能正确实时渲染 |
| AC-3 | 异常处理          | 流式传输中断时，已输出内容保留，显示友好的错误提示             |
| AC-4 | 状态管理          | 流式输出期间，发送按钮处于 loading 状态，防止重复提交       |
| AC-5 | 结束标识          | 流式输出完成后，UI 状态正确恢复（按钮可用、滚动到底部等）        |
| AC-6 | 对话存储          | 流式输出完成后，完整回答仍需正确存入数据库（保持现有逻辑不变）       |

### 2.4 技术约束

- 后端需使用 FastAPI 的 `StreamingResponse` 或 `EventSourceResponse`
- 前端需兼容现有 Axios 请求封装，或新增独立的流式请求工具函数
- 不得破坏现有的对话历史存储逻辑（完整回答仍需存入数据库）
- 流式传输协议建议采用 SSE（Server-Sent Events），实现简单且兼容性好

### 2.5 参考实现思路

**后端**：

```python
from fastapi.responses import StreamingResponse

async def stream_chat(question: str):
    async def generate():
        async for chunk in llm.astream(prompt):
            yield f"data: {chunk.content}\n\n"
        yield "data: [DONE]\n\n"
    
    return StreamingResponse(generate(), media_type="text/event-stream")
```

**前端**：

```typescript
const eventSource = new EventSource(`/api/chat/stream?question=${encodeURIComponent(question)}`)
eventSource.onmessage = (event) => {
  if (event.data === '[DONE]') {
    eventSource.close()
    return
  }
  answerContent.value += event.data
}
```

## 三、需求二：文档批量上传优化

### 3.1 问题详细描述

**当前行为**：

```
点击上传 → 选择单个文件 → 等待上传完成 → 重复以上步骤
```

**问题影响分析**：

| 影响维度 | 具体表现                        |
| ---- | --------------------------- |
| 操作效率 | 上传 10 份试卷需要重复操作 10 次，耗时且易出错 |
| 用户疲劳 | 大量重复点击操作导致用户体验下降            |
| 场景覆盖 | 无法满足「一次性上传整套复习资料」的实际使用场景    |

### 3.2 核心需求描述

扩展文档上传能力，支持多文件同时上传和文件夹一键上传。

**功能层级**：

| 优先级 | 功能      | 描述                                  |
| --- | ------- | ----------------------------------- |
| P0  | 多文件选择上传 | 支持在文件选择对话框中按住 Ctrl/Shift 多选文件，一次性上传 |
| P1  | 拖拽多文件上传 | 支持将多个文件拖拽到上传区域，批量触发上传               |
| P2  | 文件夹上传   | 支持选择整个文件夹，自动遍历文件夹内所有支持格式的文件并上传      |

**技术实现要点**：

| 层级    | 实现要求                                                                              |
| ----- | --------------------------------------------------------------------------------- |
| 前端    | `<input type="file" multiple>` 实现多选；`<input type="file" webkitdirectory>` 实现文件夹选择 |
| 前端 UI | 显示上传队列列表，展示每个文件的上传进度和状态                                                           |
| 后端    | 现有单文件上传接口无需改动，前端循环调用即可                                                            |
| 并发控制  | 限制同时上传数量（建议 3-5 个），避免服务器压力过大                                                      |

### 3.3 验收标准

| 编号   | 验收项     | 通过条件                                           |
| ---- | ------- | ---------------------------------------------- |
| AC-1 | 多文件选择   | 文件选择对话框支持多选，可同时选择多个文件                          |
| AC-2 | 批量上传    | 选中的多个文件依次/并发上传，每个文件独立显示进度                      |
| AC-3 | 上传队列 UI | 显示待上传、上传中、已完成、失败等状态的文件列表                       |
| AC-4 | 错误处理    | 单个文件上传失败不影响其他文件，失败文件可单独重试                      |
| AC-5 | 文件夹上传   | 选择文件夹后，自动识别并上传其中所有支持格式的文件（.pdf/.docx/.txt/.md） |
| AC-6 | 格式过滤    | 上传文件夹时，自动跳过不支持的文件格式，不报错                        |
| AC-7 | 并发控制    | 同时上传文件数不超过 5 个，超出部分排队等待                        |

### 3.4 技术约束

- 支持的文件格式不变：`.pdf`、`.docx`、`.doc`、`.txt`、`.md`
- 单文件大小限制不变：50MB
- 文件夹上传时需递归遍历子目录
- 上传队列需支持「全部暂停」「全部取消」操作

### 3.5 参考实现思路

**多文件选择**：

```vue

  multiple
  :auto-upload="false"
  :on-change="handleFilesSelected"
  accept=".pdf,.docx,.doc,.txt,.md"
>
  <el-button type="primary">选择文件</el-button>
</el-upload>
```

**文件夹上传**：

```vue

  type="file"
  webkitdirectory
  @change="handleFolderSelected"
/>
```

**并发控制**：

```typescript
async function uploadQueue(files: File[], maxConcurrent = 3) {
  const queue = [...files]
  const running: Promise<void>[] = []
  
  while (queue.length > 0 || running.length > 0) {
    while (running.length < maxConcurrent && queue.length > 0) {
      const file = queue.shift()!
      const promise = uploadSingleFile(file).then(() => {
        running.splice(running.indexOf(promise), 1)
      })
      running.push(promise)
    }
    await Promise.race(running)
  }
}
```

## 四、优先级与实施建议

| 需求   | 优先级 | 建议实施顺序 | 理由                     |
| ---- | --- | ------ | ---------------------- |
| 流式输出 | P0  | 先实施    | 影响所有问答场景，用户体验提升最明显     |
| 批量上传 | P1  | 后实施    | 影响文档管理场景，用户可通过分批上传临时绕过 |

## 五、非功能性要求

| 维度  | 要求                               |
| --- | -------------------------------- |
| 兼容性 | 流式输出需兼容 Chrome、Edge、Firefox 最新版本 |
| 性能  | 流式输出不应明显增加服务器 CPU/内存占用           |
| 稳定性 | 批量上传 20 个文件时，系统响应时间不超过 2 秒       |
| 可回滚 | 每项优化应能通过配置开关禁用，回退到原有行为           |
