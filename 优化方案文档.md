## 二、问题清单与优化方案

### 模块一：文档导入与解析

#### 问题 1.1 — 文档上传进度缺失
**问题描述**: 上传大文件(PDF/Word)时，后台解析过程阻塞了API请求，前端无法获取实时进度，用户不知道上传是否成功。  
**影响等级**: 高  
**涉及文件**:
- 后端: `backend/app/api/documents.py` (后台异步处理逻辑)
- 前端: `frontend/src/views/Upload.vue` (上传组件)  
**优化方案**:
1. 后端采用异步任务队列处理文档解析，解析完成后通过轮询接口返回进度
2. 前端显示进度条，包含"上传中 → 解析中 → 向量化中 → 完成"阶段提示
3. 解析失败时提供"重新解析"按钮

```python
# 后端示例：进度查询接口
@app.get("/api/documents/{doc_id}/progress")
async def get_doc_progress(doc_id: str):
    """返回当前文档解析进度"""
    doc = await db.query(Document).filter(Document.id == doc_id).first()
    return {
        "status": doc.status,           # pending/processing/completed/failed
        "progress": doc.progress,       # 0-100
        "chunk_count": doc.chunk_count, # 已分块数量
        "error": doc.error_message      # 失败原因（如有）
    }
```

#### 问题 1.2 — 上传失败无重试机制
**问题描述**: 解析中途出错只能手动重新上传，用户需要重新选择文件，体验差。  
**影响等级**: 高  
**优化方案**:
1. 文档状态表中增加 `error_message` 字段记录失败原因
2. 前端在"失败"状态显示错误信息，并提供"重新解析"按钮
3. 提供批量上传功能（需求F1.1.4定义为P1但未实现）

#### 问题 1.3 — 分块预览功能未实现
**问题描述**: 上传后无法预览分块结果，用户不知道文档是否被正确解析为题目片段。  
**影响等级**: 中  
**需求编号**: F1.2.5 (P1)  
**涉及文件**:
- 后端: `backend/app/api/documents.py` (`GET /{doc_id}/chunks` 已存在)
- 前端: `frontend/src/components/upload/ChunkPreview.vue` (组件已存在)  
**优化方案**:
1. 前端在上传完成后自动调用 chunks 接口，以可折叠列表展示每个分块
2. 每个分块显示：题号、内容摘要、来源页码
3. 支持用户手动编辑分块内容或调整分块边界

#### 问题 1.4 — 文档去重/增量更新缺失
**问题描述**: 同一份文档多次上传会产生重复的向量数据。  
**影响等级**: 中  
**优化方案**:
1. 上传时计算文件MD5哈希值，与已有文档对比
2. 若哈希值相同，提示"该文件已存在"并跳过解析
3. 哈希值不同但文件名相同，覆盖旧数据并清理旧向量

### 模块二：AI问答

#### 问题 2.1 — 响应时间过长无加载提示
**问题描述**: 每次问答需要检索知识库(30s) + LLM生成(10-30s)，用户等待时间可达40-60秒，期间无任何反馈，用户以为系统卡死。  
**影响等级**: 高  
**涉及文件**:
- 后端: `backend/app/api/chat.py` (流式响应)
- 前端: `frontend/src/views/Chat.vue` (对话界面)  
**优化方案**:
1. 检索阶段显示加载动画和"正在检索知识库..."提示
2. LLM生成阶段显示"正在思考..." + 流式输出
3. 考虑对常见公考问题做缓存（如"什么是行测""申论写作技巧"），避免重复检索

```typescript
// 前端示例：问答状态管理
const askStatus = ref<'idle' | 'searching' | 'thinking' | 'done'>('idle')

// 检索阶段
askStatus.value = 'searching'
const sources = await api.searchKnowledge(question)

// 生成阶段
askStatus.value = 'thinking'
const response = await api.streamChat(question, sources)
askStatus.value = 'done'
```

#### 问题 2.2 — 检索结果无相关性评分阈值
**问题描述**: 返回所有搜索结果，无论相似度多低，LLM可能基于无关内容给出误导性回答。  
**影响等级**: 高  
**涉及文件**: `backend/app/core/rag/retriever.py`  
**优化方案**:
1. 设置相似度阈值（如 cosine > 0.7），低于阈值的片段不参与回答生成
2. 若所有片段相似度均低于阈值，提示"知识库中暂无相关内容，基于通用知识回答"

#### 问题 2.3 — 对话标题不可修改
**问题描述**: 创建对话时标题自动从第一条用户消息截取，没有提供修改对话标题的接口。  
**影响等级**: 低  
**优化方案**:
1. 新增 `PUT /api/chat/conversations/{id}` 接口，支持修改标题
2. 前端在对话列表提供右键菜单"重命名"选项

### 模块三：题目分类

#### 问题 3.1 — LLM分类失败无fallback
**问题描述**: 规则分类命中率仅60-70%，剩余靠LLM分类，但LLM调用失败时分类功能完全失效，返回"未分类"。  
**影响等级**: 高  
**涉及文件**: `backend/app/api/classify.py`, `backend/app/core/ai/classifier.py`  
**优化方案**:
1. LLM分类失败时二次尝试规则分类（即使置信度较低）
2. 规则分类与LLM分类并行执行，取高置信度结果
3. 提供分类置信度阈值配置（如 < 0.7 时提示用户确认）

```python
# 优化后的分类逻辑
async def classify_question(text: str) -> ClassificationResult:
    # 1. 先执行规则分类
    rule_result = classify_by_rules(text)
    
    # 2. 并行执行LLM分类
    try:
        llm_result = await classify_by_llm(text)
        # 取高置信度结果
        if llm_result.confidence > rule_result.confidence:
            return llm_result
        return rule_result
    except Exception:
        # LLM失败时回退到规则分类
        return rule_result
```

#### 问题 3.2 — 分类结果无法批量修正
**问题描述**: 分类错误时只能逐题调整，效率低。  
**影响等级**: 中  
**优化方案**:
1. 新增 `PUT /api/classify/batch` 接口，支持批量修改分类
2. 前端在对话界面提供"归类错误"按钮，点击后弹出分类选择器
3. 修正后的分类同步更新到数据库和思维导图

### 模块四：思维导图

#### 问题 4.1 — 导图渲染性能差
**问题描述**: 全量加载所有节点，当题目数量超过200时渲染卡顿；`get_node_level` 存在N+1查询问题。  
**影响等级**: 高  
**涉及文件**:
- 后端: `backend/app/core/mindmap/updater.py`
- 前端: `frontend/src/components/mindmap/MindMapNode.vue`  
**优化方案**:
1. 初次只加载一层节点，用户展开时懒加载子节点
2. 数据库层面用 `selectinload` 一次性加载整棵树，消除N+1查询
3. 提供"全部展开/折叠"按钮，默认只显示一级和二级模块

```python
# 优化后：一次性加载子树
async def get_children_nodes(mindmap_id: str, parent_id: str = None):
    """只查询指定父节点的直接子节点（用于懒加载）"""
    stmt = select(MindMapNode).where(
        MindMapNode.mind_map_id == mindmap_id,
        MindMapNode.parent_id == parent_id
    )
    result = await db.execute(stmt)
    return result.scalars().all()
```

#### 问题 4.2 — 掌握度计算逻辑不透明
**问题描述**: 定义了掌握度0-100，但未明确计算方法，用户不信任这个指标。  
**影响等级**: 中  
**优化方案**:
1. 明确定义: 掌握度 = 做过该节点题目的数量 / 该类型总题量 × 正确率权重
2. 节点 hover 时显示计算依据："已做8题, 正确率75%"
3. 支持用户手动调整掌握度（做过后标记"已掌握"/"需复习"）

```python
# 掌握度计算逻辑
async def calculate_mastery(module_id: str) -> int:
    """掌握度 = 正确题目数 / 总答题数 × 100"""
    total = await db.query(
        func.count(Question.id)
    ).filter(
        Question.module_id == module_id
    ).scalar()
    
    correct = await db.query(
        func.count(Question.id)
    ).filter(
        Question.module_id == module_id,
        Question.mastery >= 70  # 掌握度>=70%视为正确
    ).scalar()
    
    return int((correct / total) * 100) if total > 0 else 0
```

### 模块七：安全与架构

#### 问题 7.1 — 无用户认证机制
**问题描述**: 任何人都可以访问系统，数据没有隔离，部署到公网存在安全风险。  
**影响等级**: 高  
**优化方案**:
1. 实现JWT Token认证（用户名密码登录）
2. 敏感API增加身份验证中间件
3. 如果是个人使用，可在设置中配置API Key保护

```python
# JWT认证中间件示例
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

security = HTTPBearer()

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security)
):
    token = credentials.credentials
    payload = decode_token(token)
    if not payload:
        raise HTTPException(status_code=401, detail="无效Token")
    return payload["user_id"]
```

#### 问题 7.2 — 无请求频率限制
**问题描述**: 所有API端点都没有速率限制，攻击者可通过高频调用耗尽LLM API配额。  
**影响等级**: 高  
**优化方案**:
1. 使用 `slowapi` 或自定义中间件实现Rate Limiting
2. 不同API设置不同限制：
   - `/api/chat`: 10次/分钟
   - `/api/documents/upload`: 5次/分钟
   - `/api/methods/generate`: 2次/小时

#### 问题 7.3 — 全局单例存在线程安全问题
**问题描述**: `vector_store.py` 和 `embedder.py` 中的全局单例在高并发下可能产生竞态条件。  
**影响等级**: 中  
**优化方案**:
1. 使用 `asyncio.Lock` 保护初始化过程
2. 或使用 `functools.lru_cache` 装饰器

```python
import asyncio
from functools import lru_cache

_chroma_lock = asyncio.Lock()

@lru_cache(maxsize=1)
async def get_vector_store():
    async with _chroma_lock:
        if _vector_store is None:
            _vector_store = Chroma(...)
        return _vector_store
```

#### 问题 7.4 — 数据库凭证硬编码
**问题描述**: `config.py` 默认密码 `2004` 硬编码在代码中。  
**影响等级**: 中  
**优化方案**:
1. 移除默认值，环境变量不存在时启动报错并提示配置
2. 提供 `.env.example` 模板文件
